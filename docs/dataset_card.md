# Dataset Card: Loan Default Risk Prediction

## Overview

This dataset contains synthetic loan application data for predicting consumer loan defaults. It includes 20,000 loan records from approximately 18,000 borrowers, with a realistic 7% default rate. The dataset is designed to evaluate machine learning systems on a realistic binary classification task with inherent complexity including class imbalance, grouped data structure, and feature interactions.

**Task Type:** Binary Classification  
**Metric:** AUC-ROC (Area Under the ROC Curve)  
**Dataset Size:** 20,000 loans (14,000 train / 6,000 test)  
**Features:** 17 predictive features  
**Target:** Loan default indicator (0 = fully paid, 1 = defaulted)  

## Generation Methodology

This dataset is **fully synthetic** and was generated programmatically using controlled statistical distributions and realistic business rules. The generation process includes:

1. **Feature Generation:**
   - Credit scores drawn from normal distribution (mean=700, std=60)
   - Income from log-normal distribution (median ~$50k)
   - Loan amounts from log-normal distribution with heavy tail
   - Categorical features sampled from realistic frequency distributions
   - Structured missingness in employment and delinquency history fields

2. **Relationship Modeling:**
   - Interest rates inversely correlated with credit scores
   - Loan amounts positively correlated with income levels
   - DTI ratios calculated from income and debt obligations
   - Credit limits scaled by income with controlled variance

3. **Target Generation:**
   - Base default probability of 7%
   - Risk factors with interaction effects:
     - Poor credit score AND high debt-to-income → elevated risk
     - Large loan relative to income → elevated risk
     - Recent delinquencies → elevated risk
     - Employment instability AND renting → elevated risk
     - Geographic risk factors (certain states)
   - Logistic function applied to composite risk score
   - Random noise added to prevent perfect predictability

4. **Train/Test Split:**
   - Split by borrower ID (not by loan ID) to prevent leakage
   - Stratified by default status to maintain class balance
   - 70/30 split ratio
   - Random seed 42 for reproducibility

The generation script is included in this repository and can be re-run to reproduce the exact dataset.

## Target Definition

**Column:** `default`  
**Type:** Binary integer (0 or 1)  
**Meaning:**
- `0` = Loan was fully paid off (successful completion)
- `1` = Loan defaulted (borrower failed to repay)

**Class Distribution:**
- Class 0 (paid): ~93%
- Class 1 (default): ~7%

The target exhibits realistic class imbalance typical of loan default prediction tasks.

## Train/Test Split Methodology

The dataset is split using a **grouped strategy** to prevent data leakage:

- **Grouping Variable:** `borrower_id`
- **Split Ratio:** 70% train / 30% test
- **Key Property:** No borrower appears in both train and test sets
- **Stratification:** Split maintains approximately equal default rates in both sets
- **Random Seed:** 42 (deterministic split)

This approach is critical because approximately 2,000 borrowers have multiple loans in the dataset. A naive random split would place loans from the same borrower in both train and test sets, allowing models to implicitly memorize borrower-level characteristics and overestimate performance.

**Train Set:**
- 14,000 loan records
- ~12,600 unique borrowers
- ~7% default rate

**Test Set:**
- 6,000 loan records
- ~5,400 unique borrowers
- ~7% default rate

## Leakage Risks

This dataset intentionally includes potential leakage risks that sophisticated ML practitioners should identify and avoid:

### 1. Borrower ID Leakage (Critical)
**Risk:** Using `borrower_id` as a feature creates direct information leakage.  
**Why:** The ID carries no predictive information about default risk, but in naive cross-validation, the model can memorize individual borrower tendencies if the same borrower appears in multiple folds.  
**Prevention:** Exclude `borrower_id` from feature set. Use GroupKFold or StratifiedGroupKFold for cross-validation.

### 2. Random Cross-Validation Leakage
**Risk:** Standard random K-fold splits will leak information for borrowers with multiple loans.  
**Why:** Loans from the same borrower may appear in both training and validation folds, allowing the model to effectively "see" the validation set through the lens of that borrower's other loans.  
**Prevention:** Always use GroupKFold with `borrower_id` as the grouping variable.

### 3. Temporal Leakage (Minor)
**Risk:** The `application_year_month` feature could theoretically introduce temporal dependencies.  
**Why:** If test set is from a later time period with different economic conditions, using application date directly might not generalize.  
**Note:** In this dataset, train/test are randomly split across time, so this risk is minimal, but practitioners should be aware of temporal validation in real-world scenarios.

### 4. Target Encoding Leakage
**Risk:** Encoding high-cardinality categoricals (like `state` or `loan_purpose`) using target statistics without proper cross-validation.  
**Why:** Computing target encodings on the full training set before CV creates look-ahead bias.  
**Prevention:** Perform target encoding within each CV fold, using only that fold's training data.

## Ethical Considerations

While this dataset is fully synthetic and contains no real individuals, it models a domain (consumer credit) with significant ethical implications in real-world applications:

1. **Fairness and Bias:** Real loan default models can perpetuate or amplify historical biases. This dataset intentionally **does not** include protected attributes like race, gender, or age. Practitioners should be aware that in real applications, careful bias auditing is required.

2. **Geographic Disparities:** The dataset includes state-level variation in default rates. While synthetic, this reflects real-world geographic economic differences. Models should be evaluated for fairness across regions.

3. **Financial Impact:** In real applications, default prediction models directly affect people's access to credit and interest rates charged. High-stakes decisions require careful validation, interpretability, and human oversight.

4. **Explainability:** Credit decisions in many jurisdictions require explainability (e.g., adverse action notices). Practitioners should consider model interpretability alongside predictive performance.

This synthetic benchmark allows testing ML capabilities without the ethical risks of using real financial data.

## Why This Dataset is Non-Trivial

This dataset is designed to resist trivial solutions and test genuine ML competence:

1. **Grouped Structure:** The presence of repeat borrowers requires proper grouped validation. Naive approaches will overfit.

2. **Class Imbalance:** 7% positive class requires appropriate handling (stratified sampling, class weights, appropriate metrics).

3. **Feature Interactions:** Default risk depends on combinations of factors (e.g., credit score AND debt-to-income) that linear models struggle to capture.

4. **High-Cardinality Categoricals:** State (50 values) and loan purpose (10 values) require thoughtful encoding strategies.

5. **Structured Missingness:** Missing employment length and delinquency history are informative signals, not random noise.

6. **Realistic Noise:** The target includes controlled noise to prevent perfect separation and test generalization.

7. **Heavy-Tailed Distributions:** Loan amounts and income have realistic skew that affects model training.

A simple baseline (logistic regression with minimal preprocessing) achieves ~0.70 AUC. Proper ML with grouped validation, careful feature engineering, and appropriate modeling should reach 0.78-0.82 AUC. The gap between naive and principled approaches is substantial.

## Suitability for ML Benchmarking

This dataset is well-suited for evaluating ML systems because it:

- **Tests validation strategy understanding:** Grouped structure punishes naive CV
- **Tests preprocessing skills:** Missing values, encoding, scaling require thoughtful handling
- **Tests modeling choices:** Tree ensembles significantly outperform linear models
- **Tests metric understanding:** AUC-ROC is appropriate for imbalanced classification
- **Tests domain reasoning:** Feature engineering benefits from financial domain knowledge
- **Resists gaming:** No shortcuts or memorization strategies work
- **Realistic scale:** 20k rows is large enough for meaningful patterns, small enough for rapid iteration

Expected performance ranges:
- Random baseline: ~0.50 AUC
- Linear model (basic preprocessing): 0.70-0.73 AUC
- Random Forest (default params): 0.74-0.77 AUC
- Tuned GBDT with proper validation: 0.78-0.82 AUC
- Perfect predictions: 1.00 AUC (Oracle)

The spread between naive and sophisticated approaches is substantial enough to differentiate ML capability levels.
